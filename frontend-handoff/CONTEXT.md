# Legal Management System — Frontend Domain Glossary

This repository is the **frontend** of the Legal Management System: a Next.js (App Router) + TypeScript application deployed on Vercel. It communicates with a separate Express backend API over HTTP. All business logic lives in the backend — this repo owns routing, UI, auth session management, and form handling.

---

## Actors & Roles

All roles share one Next.js application with role-based routing. Each role maps to a distinct section of the app.

**IT Admin**
Superadmin. Accesses `/admin`. Manages users, roles, SLA config, holidays, dropdown options (Lokasi Kantor, Divisi, Unit Bisnis), and Vendor accounts.

**Requestor**
Internal school employee. Accesses `/dashboard`. Submits and tracks legal requests. Must complete profile on first login before accessing any other page.

**Legal Team**
Accesses `/legal`. Processes the shared request queue. No profile completion required.

**Vendor**
External counterparty. Accesses `/vendor`. Completes KYB on onboarding, updates documents, and tracks the status of requests they are linked to. Cannot sign documents through the portal.

---

## Authentication

- Provider: **Supabase Auth** (email + password)
- The frontend reads the Supabase JWT and attaches it to every API request (`Authorization: Bearer <token>`)
- Role is stored in the JWT claims (set by the backend/Supabase on account creation)
- **Vendor onboarding:** Vendor receives an email invitation and sets their password on first login
- **Requestor first login:** After authentication, if profile is incomplete, redirect to `/onboarding` before allowing any navigation

---

## Requestor Profile Completion Gate

On first login, Requestor is redirected to `/onboarding` and must complete:
- Nama Lengkap
- Lokasi Kantor (dropdown)
- Divisi (dropdown)
- Unit Bisnis (dropdown)

Profile can be updated at any time from account settings. Changes apply to new requests only — existing reference numbers are not retroactively updated.

## LegalRequest

The central entity. Submitted by a Requestor, processed by Legal Team through a defined workflow.

**Reference Number:** Auto-generated by the backend on submission.
Format: `{SEQUENCE}/{DIVISI_CODE}/{TYPE_CODE}/{MONTH_ROMAN}/{YEAR}`
Example: `042/FIN/PB/V/2026`

### Request Types

| Type | Indonesian Name | Vendor Required |
|------|----------------|-----------------|
| `PERJANJIAN_BARU` | Pembuatan/Review Perjanjian Baru | Yes |
| `ADENDUM` | Pembuatan/Review Adendum Perjanjian | Yes |
| `SURAT` | Pembuatan/Review Surat | No |
| `PERMINTAAN_DOKUMEN` | Permintaan Dokumen | No |

### Submission Flow

**PERJANJIAN_BARU and ADENDUM** (multi-step form):
1. Select request type
2. Search and link a Vendor by name — system shows matching vendors with KYB status
   - If not found: enter vendor email → system sends onboarding invitation
3. Fill in form fields
4. Save as draft or submit

**SURAT and PERMINTAAN_DOKUMEN** (single-step form):
1. Select request type
2. Fill in form fields
3. Save as draft or submit

### Form Fields per Type

**PERJANJIAN_BARU**
- Lingkup Perjanjian (text)
- Status Perjanjian (enum: `BELUM_BERLANGSUNG` / `SEDANG_BERLANGSUNG` / `SUDAH_SELESAI`)
- Jangka Waktu Perjanjian (date range)

**ADENDUM**
- Perjanjian sebelumnya yang hendak diubah (free-text)
- Hal yang ingin diubah (text)
- Lampirkan Perjanjian Sebelumnya (file upload)

**SURAT**
- Surat yang hendak dibuat (string)
- Identitas Penerima Surat (string)
- Korespondensi surat sebelumnya (file upload, optional)

**PERMINTAAN_DOKUMEN**
- Dokumen Perusahaan yang diminta (string)
- Tujuan Permintaan Dokumen (text)
- Lampiran Pendukung (file upload, optional)

### Draft

A request can be saved as a draft at any point before submission. Drafts appear in the Requestor's dashboard with a "Draft" badge. Drafts have no reference number.

## Vendor

A first-class entity with their own account and KYB status.

**Types:** Badan (corporate) / Perorangan (individual)

**KYB Status** (visible to Requestor when searching, and to Legal Team):

| Status | Label shown in UI |
|---|---|
| `INVITED` | Menunggu Pendaftaran |
| `SUBMITTED` | Menunggu Verifikasi |
| `REVISION` | Perlu Revisi |
| `APPROVED` | Terverifikasi |

### Vendor Portal (`/vendor`)

Pages available to Vendor:
- **KYB submission** — upload required documents based on their type (Badan/Perorangan)
- **Document management** — view and update previously submitted KYB documents
- **Request tracker** — view status of all requests they are linked to (read-only)

Vendors cannot sign documents through the portal. Signing is handled offline.

## Workflow Stages

### Stage Definitions

| Stage | Key | Visible label |
|-------|-----|---------------|
| Menunggu Respon | `WAITING` | Menunggu Respon (or "Menunggu KYB Vendor" if vendor KYB is incomplete) |
| Legal Review | `LEGAL_REVIEW` | Sedang Diproses |
| User Review | `USER_REVIEW` | Perlu Revisi |
| Vendor Review | `VENDOR_REVIEW` | Menunggu Konfirmasi Vendor |
| Internal Signing | `INTERNAL_SIGNING` | Penandatanganan Internal |
| Vendor Signing | `VENDOR_SIGNING` | Penandatanganan Vendor |
| Finished | `FINISHED` | Selesai |
| Cancelled | `CANCELLED` | Dibatalkan |
| Rejected | `REJECTED` | Ditolak |

### Stage Paths per Request Type

| Request Type | Happy path |
|---|---|
| `PERJANJIAN_BARU` | `WAITING → LEGAL_REVIEW → VENDOR_REVIEW → INTERNAL_SIGNING → VENDOR_SIGNING → FINISHED` |
| `ADENDUM` | `WAITING → LEGAL_REVIEW → VENDOR_REVIEW → INTERNAL_SIGNING → VENDOR_SIGNING → FINISHED` |
| `SURAT` | `WAITING → LEGAL_REVIEW → [INTERNAL_SIGNING →] FINISHED` |
| `PERMINTAAN_DOKUMEN` | `WAITING → LEGAL_REVIEW → [INTERNAL_SIGNING →] FINISHED` |

`USER_REVIEW` is entered only when Legal Team sends the request back for revision (not on the happy path). `INTERNAL_SIGNING` is optional for SURAT and PERMINTAAN_DOKUMEN.

### Requestor Actions by Stage

| Stage | Requestor can |
|---|---|
| `WAITING` | View status, cancel request |
| `USER_REVIEW` | Edit request fields, resubmit, cancel |
| All others | View status only |

### Legal Team Actions by Stage

| Stage | Legal Team can |
|---|---|
| `WAITING` | Advance to `LEGAL_REVIEW` (only if vendor KYB is `APPROVED`) |
| `LEGAL_REVIEW` | Advance to next stage, send back to `USER_REVIEW` (with remarks), decide if `INTERNAL_SIGNING` needed (for SURAT/PDK), reject |
| `USER_REVIEW` | — (waiting for Requestor) |
| `VENDOR_REVIEW` | Click "Vendor Confirmed" to advance, send back to `USER_REVIEW`, reject |
| `INTERNAL_SIGNING` | Advance to next stage, send back to `USER_REVIEW`, reject |
| `VENDOR_SIGNING` | Advance to `FINISHED`, send back to `USER_REVIEW`, reject |
| `FINISHED` | — (terminal) |

To reach `FINISHED`, Legal Team uploads final signed document(s) (multiple files, max 20MB each).

### Cancellation & Rejection

- **Cancelled:** Requestor-initiated (from `WAITING` or `USER_REVIEW`) or Legal Team-initiated (any stage). Record preserved in history.
- **Rejected:** Legal Team-initiated (any stage). Mandatory reason required. Requestor notified by email. Cannot be resubmitted — Requestor opens a fresh request.

## SLA

Displayed on each request card and detail page. SLA clock starts at submission.

- Default: 5 working days (excludes weekends, national holidays, custom off-days)
- States: On Track / Approaching (≤1 day remaining) / Breached
